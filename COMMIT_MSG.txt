Fix context-path bugs breaking the app under a Tomcat context

After login the app redirected to /dashboard and returned 404: under Tomcat
it runs in the /aquariusweb context, but several absolute paths ignored it.

- SecurityConfig: the login success handler used
  res.sendRedirect("/dashboard"); HttpServletResponse.sendRedirect does not
  prepend the context path. Now uses req.getContextPath() + "/dashboard".
- Four JS fetch() calls used absolute URLs and would hit the server root
  instead of the app: the sidebar menu loader (/api/menu/tree), custom-report
  validate and execute, and the help chat. The menu loader now reads a new
  <meta name="_ctx"> exposed by the layout; the three th:inline templates use
  the Thymeleaf /*[[@{/...}]]*/ expression so the URL includes the context.

Controller "redirect:/..." returns and Thymeleaf @{...} links were already
correct (Spring/Thymeleaf add the context), as was FiscalContextInterceptor.

Verification: Java brace scan, node --check on the inline-script templates,
no remaining context-less absolute fetch in templates, and git apply --check
clean on a fresh clone of HEAD e67cd01. Live login flow to confirm on deploy.
